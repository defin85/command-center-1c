  Query Parameters:

    - offset (default: 0)
  Response: { executions: [...], count: N, total: N }
  Features:

    * Status enum validation
    * Pagination with safe limit/offset parsing
    * Database optimization with select_related()
    * Audit logging with user context

Endpoint 2: GET /api/v2/workflows/get-execution/?execution_id=X
  Function: get_execution(request)
  Purpose: Get detailed execution information with steps
  Query Parameters:
    - execution_id (required): Execution UUID
  Response: { execution: {...}, steps: [...] }
  Features:
    * UUID validation
    * 404 handling for non-existent executions
    * Prefetch optimization for step_results
    * Full execution context with step history

Endpoint 3: POST /api/v2/workflows/cancel-execution/
  Function: cancel_execution(request)
  Purpose: Cancel pending or running execution
  Request Body: { execution_id: "uuid" }
  Response: { execution_id: ..., cancelled: true, status: "cancelled" }
  Features:
    * UUID validation
    * FSM state machine integration
    * Status validation (only pending/running can be cancelled)
    * Proper error messages for invalid transitions
    * Audit logging

Endpoint 4: GET /api/v2/workflows/get-execution-steps/?execution_id=X
  Function: get_execution_steps(request)
  Purpose: Get all step results for execution
  Query Parameters:
    - execution_id (required): Execution UUID
    - status (optional): Filter by step status
  Response: { steps: [...], count: N }
  Features:
    * UUID validation
    * Step status filtering with validation
    * Supports all step statuses (pending, running, completed, failed, skipped)
    * Ordered by started_at for timeline
    * Execution existence verification
