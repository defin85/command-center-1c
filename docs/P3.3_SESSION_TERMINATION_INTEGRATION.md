# P3.3 Session Termination Integration

**Status:** ✅ Completed
**Date:** 2025-11-09
**Phase:** Track 4 - Batch Service Advanced Features

---

## Обзор

Реализована интеграция с cluster-service для автоматического завершения активных сеансов 1С перед установкой расширений.

**Проблема:** Активные сеансы пользователей блокируют установку расширений в базы 1С.

**Решение:** Автоматическое завершение сеансов через cluster-service API с retry logic и graceful timeout.

---

## Архитектура решения

### Integration Flow

```
1. Django → batch-service: POST /api/v1/extensions/install
   {
     "force_terminate_sessions": true,  // NEW optional parameter
     ...
   }

2. batch-service проверяет force_terminate_sessions

3. Если true:
   a. batch-service → cluster-service: GET /api/v1/sessions?infobase_id={uuid}
   b. Если есть активные сеансы:
      - batch-service → cluster-service: POST /api/v1/sessions/terminate
        {infobase_id, session_ids: [...]}
      - Grace period: 7 секунд
      - Retry проверка: до 3 попыток с интервалом 3 сек
   c. Если сеансы не завершились → return error

4. batch-service → 1cv8.exe: Install extension

5. Результат → Django callback
```

### Компоненты

**batch-service:**
- `internal/models/cluster.go` - Session models
- `internal/infrastructure/cluster/client.go` - HTTP client для cluster-service
- `internal/domain/session/manager.go` - Бизнес-логика с retry logic
- `internal/service/extension_installer.go` - Интеграция в installer
- `internal/config/config.go` - Новый параметр ClusterServiceURL

**cluster-service:**
- `internal/models/session.go` - Session models
- `internal/api/handlers/sessions.go` - MOCK endpoints (для P3.3)
- `internal/api/router.go` - Новые routes

## Ключевые технические решения

### 1. Infrastructure Layer - HTTP Client

**Файл:** `go-services/batch-service/internal/infrastructure/cluster/client.go`

**Особенности:**
- HTTP timeout: 30 секунд
- Thread-safe (http.Client)
- Детальное логирование всех запросов
- Graceful error handling

**Методы:**
- `GetSessions(infobaseID string) ([]models.Session, error)`
- `TerminateSessions(infobaseID string, sessionIDs []string) (*models.TerminateSessionsResponse, error)`
- `HealthCheck() error` - non-blocking проверка при старте

### 2. Domain Layer - SessionManager

**Файл:** `go-services/batch-service/internal/domain/session/manager.go`

**Retry Logic:**
```go
Grace period: 7 секунд после terminate
Max retries: 3 попытки проверки
Retry interval: 3 секунды между попытками
Total timeout: ~50 секунд максимум
```

**Обработка ошибок:**
```go
var (
    ErrClusterServiceUnavailable = errors.New("cluster-service unavailable")
    ErrSessionsNotTerminated     = errors.New("failed to terminate all sessions")
    ErrSessionCheckFailed        = errors.New("failed to check session status")
)
```

**Алгоритм:**
1. Проверка `force` параметра (skip если false)
2. Получение списка активных сеансов
3. Если сеансов нет → success (skip)
4. Terminate request → cluster-service
5. Grace period (7 sec)
6. Retry verification loop (до 3 попыток)
7. Success если все сеансы завершены, Error если остались активные

### 3. Integration в ExtensionInstaller

**Изменения:**
- Добавлен `sessionManager *session.SessionManager` в структуру
- Обновлен конструктор для принятия sessionManager и logger
- Добавлен блок session termination ПЕРЕД установкой расширения

**Flow:**
```go
1. Log start installation
2. IF force_terminate_sessions:
   - Call sessionManager.TerminateSessionsIfNeeded()
   - IF error → return error (блокируем установку)
3. Create infobase connection
4. Load extension
5. Execute installation
6. Update DB config (if requested)
```

### 4. Configuration

**Новая переменная окружения:**
```bash
CLUSTER_SERVICE_URL=http://localhost:8088  # default
```

**Health check при старте:**
- Non-blocking (warning log если недоступен)
- Позволяет batch-service работать без cluster-service
- Session termination features будут unavailable до появления cluster-service

## Известные ограничения

### 1. MOCK Implementation (P3.3)

**Что НЕ реализовано:**
- Реальная интеграция с RAS gRPC
- Получение реальных сеансов из 1С кластера
- Реальное завершение сеансов через RAS

**Зачем MOCK:**
- Проверить корректность integration flow
- Протестировать retry logic
- Убедиться что error handling работает
- Готовность к реальной интеграции в будущем

### 2. Infobase ID Resolution

**Текущая реализация:**
```go
// TODO: Get infobase UUID from infobase name (requires cluster-service integration)
// For now, use infobase name as ID (MOCK implementation)
infobaseID := req.InfobaseName
```

**Проблема:** В реальной системе нужен UUID информационной базы, а не имя.

**Решение для Production:**
- Добавить endpoint в cluster-service: `GET /api/v1/infobases/resolve?name={name}` → UUID
- Или передавать UUID из Django (если база уже зарегистрирована)

### 3. Graceful Degradation

**Поведение при недоступности cluster-service:**
- batch-service стартует успешно (warning log)
- Session termination features unavailable
- Установка БЕЗ force_terminate_sessions работает нормально
- Установка С force_terminate_sessions → error

**Альтернатива (обсудить):**
- Опция: Игнорировать ошибку cluster-service и продолжить установку (risky)
- Current: Strict mode (fail-fast если не удалось завершить сеансы)

### 4. Concurrent Requests

**Thread-safety:**
- ClusterClient: thread-safe (http.Client)
- SessionManager: thread-safe (stateless)
- ExtensionInstaller: thread-safe

**Connection limits:**
- HTTP client: 30 sec timeout per request
- Concurrent termination requests: unlimited (controlled by HTTP client pool)
