# Roadmap: UI + Operations v2 (Streams-first)

Цель: привести UI и операции к единому и прозрачному lifecycle на Redis Streams, унифицировать управление базами/кластерами, и убрать устаревшие сценарии/экраны.

Фокус: фронтенд (SPA), Orchestrator API, Worker lifecycle/streams, согласованная модель данных.

---

## Входные проблемы (кратко)

- Неполные параметры для блокировки начала сеансов.
- Нет явных индикаторов состояний баз (RAS-сессии, регламентные задания).
- Health check не прозрачен и даёт “degraded” без объяснений.
- Несогласованные логины/пароли кластеров и баз.
- Операции и трассировка разбросаны по разным экранам.
- Прогресс операций и задачи не обновляются.
- “New Operation” не даёт выбрать все операции и не масштабируется.
- Управление расширениями не унифицировано со Streams, есть 500/401 ошибки.
- Нет операций для новых драйверов на карточке базы.
- Нет редактирования шаблонов из UI.
- Нет архитектуры i18n.
- Нет единого трейсинга операций/workflow (live + history).

---

## Ограничения и принципы

- **OpenAPI contract-first**: любые API изменения фиксируются в `contracts/**`.
- **Streams-first**: прогресс/статусы операций и health должны идти через Streams.
- **Безопасность**: пароли не возвращаются в UI, только флаги наличия.
- **1C транзакции < 15s**, долгие операции идут через Worker.
- **Coverage > 70%** на изменяемые модули.

## Декомпозиция на атомарные этапы

### Этап 0 — Инвентаризация и контракты

Результат:
- Реестр всех типов операций и их драйверов (Worker).
- Модель данных статусов БД (scheduled_jobs_deny, sessions_deny, denied_*).
- Перечень требуемых API изменений и связанный список контрактов в `contracts/**`.

Проверка:
- Документ с матрицей “операция → драйвер → payload → UI”.
- Список контрактов с точными полями и HTTP-эндпоинтами.

### Этап 1 — Блокировка начала сеансов: все параметры

Результат:
- UI форма “Block Sessions” собирает `denied_from`, `denied_to`, `message`, `permission_code`, `parameter`.
- Payload в операции содержит все поля, валидируется на фронте и бэке.
- Контракт обновлён (если нужно).
Статус: ✅ выполнено

Проверка:
- Операция отправляется со всеми параметрами и сохраняется в истории.
- Worker получает корректный payload (логи/trace).

---

### Этап 2 — Статусы баз в списке `/databases`

Результат:
- API для списка БД возвращает статусы RAS: `scheduled_jobs_deny`, `sessions_deny`, `denied_from/to`, `denied_message`.
- Состояния обновляются после `sync_cluster` и после операций RAS.
- UI отображает состояния в списке (теги/иконки + hover), без изменения существующих колонок `Status` и `Health`.
Статус: ✅ выполнено

Проверка:
- После lock/unlock статусы визуально меняются без ручного refresh.
- После `sync_cluster` статусы корректны для всех баз.
- Колонки `Status` и `Health` сохраняют текущую семантику и отображение.

---

### Этап 2.1 — Унификация семантики `Status` и `Health`

Результат:
- `Status` = операторский статус (active/inactive/maintenance) и меняется только пользовательскими действиями.
- `Health` = техническое состояние (ok/degraded/down/unknown), обновляется **только** health_check через Streams.
- Единые правила источника/обновления и согласованные цвета/иконки/tooltip.
- Добавлен краткий сводный бейдж состояния рядом с названием БД (не заменяет колонки).
Статус: ✅ выполнено

Проверка:
- `Status` не меняется от health_check.
- `Health` не меняется от ручного смены `Status`.
- Визуальные обозначения совпадают по всему UI.

---

### Этап 3 — Health check через Streams

Результат:
- Кнопка Health Check инициирует operation через Streams (не прямой REST).
- Прогресс и результат приходят через timeline/streams.
- UI показывает причину статуса (ок/ошибка/таймаут).
Статус: ✅ выполнено

Проверка:
- В timeline видны этапы health_check.
- Статус БД меняется только на основании результата операции.

### Этап 4 — Унификация логина/пароля (кластер + база)

Результат:
- Единый контракт для установки/обновления логина и пароля.
- UI показывает “Configured” вместо подстановки дефолтов.

Статус: ✅ выполнено

Проверка:
- После сохранения UI показывает “Configured”.
- Пароли не возвращаются в API.
- Операции используют сохранённые креды.

### Этап 5 — Единый экран операции + трассировка

Результат:

- `/service-mesh` использует те же компоненты и даёт переход в детали.
- Live Monitor либо удалён, либо заменён на streams-based timeline.
- Добавлены поля корреляции `workflow_execution_id`/`node_id`/`trace_id` в событиях и UI.
Статус: ✅ выполнено

Проверка:
- Одна операция отображается одинаково в обоих разделах.
- Нет “пустых” блоков и устаревших статусов.

### Этап 6.5 — Унификация табличного представления (пилот: databases/clusters)

Результат:
- Единый Table Toolkit (поиск, фильтры, сортировка, пагинация, пресеты).
- Серверные фильтры/сортировка/пагинация на `/databases` и `/clusters`.
- UI-метаданные таблиц через `/api/v2/ui/table-metadata/` (колонки, группы, фильтры).
- Группировка заголовков колонок (multi-header).
- Строка фильтров отдельной строкой над таблицей, выровнена по колонкам; скрытие фильтра запрещено, если есть значение.
- Настройки таблицы с пресетами в `localStorage` (видимость/порядок колонок, фильтры, сортировки).
Статус: ✅ выполнено

Проверка:
- На страницах `/databases` и `/clusters` работает полный набор: поиск, фильтры, сортировка, пагинация.
- Фильтры синхронизированы с колонками и не “уезжают” при горизонтальном скролле.
- Переключение видимости фильтра работает через иконки в заголовках.

Замечания ревью (закрыто, проверено 2026-01-19):
- ✅ `credentials`/`restrictions`: `server_field` совпадает с ключами серверной фильтрации (не приводит к `UNKNOWN_FILTER`).
- ✅ Строка фильтров синхронизирована с горизонтальным скроллом таблицы.
- ✅ Типы фильтров для чисел/дат заданы как `number`/`date` (есть `before/after`, UI не шлёт несовместимые значения).
- ✅ Пресеты в `localStorage` нормализуются при изменении метаданных и не остаются на фолбэк-конфигурации.

Дальше (следующий шаг):
- Масштабирование на остальные таблицы (operations, jobs, templates, etc.).
- Доработка UX: группировка колонок, управление фильтрами/сортировкой через пресеты.

### Этап 9 — Расширения: полный lifecycle через Streams

Результат:
- Полностью удалён runtime Action Catalog (`ui.action_catalog`) как execution source.
- Введён templates-only manual operations flow:
  - `manual_operation` + template-based resolve;
  - tenant preferred bindings для `extensions.sync` и `extensions.set_flags`;
  - fail-closed ошибки при отсутствии/устаревании binding.
- UI `/databases`:
  - single DB и bulk сценарии работают через единый plan/apply pipeline.
- API `GET /api/v2/ui/action-catalog/` переведён в стабильный decommission-контракт `404 (NOT_FOUND)`.

Проверка:
- Планирование/применение manual operations проходит через `POST /api/v2/extensions/plan|apply`.
- Для `extensions.sync` snapshot расширений сохраняется в Postgres и читается через `get-extensions-snapshot`.

### Этап 9.1 — Расширения: общий обзор `/extensions`

Результат:
- Добавлен экран `/extensions` (таблица “расширение ↔ базы”) поверх сохранённых snapshot'ов (без live-обхода всех баз).
- Snapshot расширений сохраняет нормализованный формат `extensions[]` (best-effort парсинг stdout) + `raw` + `parse_error` для диагностики.
- Добавлены API v2:
  - `GET /api/v2/extensions/overview/` — агрегированная таблица (installed/active/inactive/missing/unknown) + поиск/фильтры + пагинация.
    - Опционально: `database_id` ограничивает *набор имён расширений в выдаче* теми, которые присутствуют в snapshot выбранной базы (агрегаты по этим расширениям считаются по всем доступным базам, как и без фильтра).
  - `GET /api/v2/extensions/overview/databases/` — drill-down до списка баз по выбранному расширению/статусу/версии.
- RBAC: ответы учитывают только базы, доступные пользователю (`view_database`).
Статус: ✅ выполнено

Проверка:
- В `/extensions` видна агрегированная таблица и работает drill-down до списка баз.
- Фильтры `search/status/cluster/version/database` корректно меняют выдачу (фильтр `database` влияет только на верхнюю агрегированную таблицу).

### Этап 14 — Мультивкладочный live-stream (варианты)

Результат:
- Зафиксировано решение по поддержке нескольких вкладок (single-SSE vs multi-SSE).
- Описаны варианты и их trade-offs:
  - Вариант A: multi-SSE на пользователя (снять ограничение, каждый tab отдельно).
  - Вариант B: server-side fan-out (один источник, раздача подписчикам).
  - Вариант C: лидер-вкладка + BroadcastChannel (один SSE, локальный fan-out).
  - Вариант D: Service Worker как hub (SSE в SW, рассылка вкладкам).
- Выбран целевой вариант + критерии (нагрузка, сложность, стабильность, UX).
- План миграции/совместимости и лимиты (max tabs, fallback).

Проверка:
- Одновременные вкладки не вызывают 429/конфликтов.
- Вкладки получают идентичные события (invalidation, status).
- При закрытии лидера/потери связи есть авто-перевыбор.

### Этап 15 — SSH-драйвер для 1С (Worker)

Результат:
- Отдельный driver в Worker для выполнения 1С-операций через SSH (Designer/CLI).
- Поддержка `ssh_host`/`ssh_credentials` в payload и валидация обязательных полей.
- Единый интерфейс DesignerClient: локальный CLI и SSH-режим за фичефлагом/конфигом.
- Логирование/трейсинг выполнения (timeline) с раздельной меткой backend-режима.

Проверка:
- Операции install/remove extension и config load/dump/update работают через SSH на тестовой ИБ.
- Ошибки SSH (auth/host key/timeout) отражаются в timeline и возвращаются в UI.
