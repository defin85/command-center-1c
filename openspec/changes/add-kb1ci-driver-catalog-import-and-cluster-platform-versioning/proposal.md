# Change: Добавить импорт driver catalogs из kb.1ci.com + версионирование платформы по кластерам

## Why
Сейчас базовые каталоги драйверов (`cli`, `ibcmd`) пополняются из ITS, что:
- требует интерактивной авторизации (ИТС) и усложняет поддержку пайплайна,
- не даёт удобного источника “честной” документации без ИТС для operator/admin,
- не покрывает будущее требование: несколько наборов документации/каталогов под разные версии платформы и автоматический выбор “правильного” набора для кластера.

`https://kb.1ci.com` доступна без авторизации ИТС и предоставляет XWiki REST API, что позволяет сделать воспроизводимую загрузку/парсинг документации и связывать её с версиями платформы.

## What Changes
- Добавляется новый doc-source `kb1ci`:
  - синхронизация дерева разделов и страниц KB через XWiki REST API (с сохранением snapshot + метаданных),
  - UI для просмотра полного дерева разделов и выбора поддеревьев (DocSet) для парсинга CLI-документации,
  - администратор вручную маппит DocSet → `driver` (`cli`/`ibcmd`) + `platform_version` (набор документации под версию платформы).
- Добавляется пайплайн “Parse & Import” для driver catalogs из `kb1ci`:
  - парсинг выбранного DocSet в `command schemas` (base catalog v2) с **строгой schema-валидацией** (запрет неожиданных полей/типов),
  - отчёт по валидации и “dry-run” до сохранения/публикации,
  - сохранение получившегося base catalog как версионированного артефакта.
- Добавляется “platform versioner” по кластерам:
  - хранение `effective_platform_version` по кластеру (auto + manual override),
  - выбор активного base catalog для каждой `platform_version` (8.3.23, 8.3.24, …) в UI,
  - возможность однозначно понимать, какой набор документации/каталога должен применяться к конкретному кластеру.

## Impact
- Affected specs (новые):
  - **NEW**: `kb1ci-doc-ingestion` (интеграция с kb.1ci.com: дерево, snapshots, DocSets)
  - **NEW**: `driver-catalog-kb1ci-import` (парсинг DocSet → base catalogs + строгая валидация)
  - **NEW**: `cluster-platform-versioning` (версионирование платформы по кластерам + выбор активных каталогов по версии)
- Affected code (ожидаемо):
  - Orchestrator: новые модели/эндпоинты для kb snapshots + DocSets + parse runs; расширение driver catalogs (выбор active по версии); хранение версии по кластеру
  - Frontend: новая UI-секция для управления `kb1ci` (дерево/DocSets/parse runs) и настройка active catalogs по версиям
  - Contracts/OpenAPI: новые `/api/v2/...` эндпоинты и типы

## Risks / Mitigations (ключевые)
- Нестабильность разметки/структуры документации (XWiki syntax `xwiki/2.1`) и возможные изменения страниц.
  - Митигация: хранить raw контент + parsed артефакт отдельно; парсинг делать воспроизводимым (pinned parser version); детектить изменения по `page.version`/`modified`.
- Частичная/ошибочная применимость документации к версиям платформы (патч-версии, “смешанные” кластеры).
  - Митигация: нормализовать версии (например `8.3.23.*` → `8.3.23`), хранить `status=ok|unknown|mixed`; при `unknown/mixed` требовать ручного override для назначения документации/каталога.
- Риск “тихой деградации” при невалидном парсинге (неожиданные поля/типы).
  - Митигация: строгая schema-валидация на этапе parse/import (fail-closed), отчёт с путями ошибок; сохранение base catalog только при `severity=error` отсутствует.
- Производительность/нагрузка на kb.1ci.com (массовые обходы дерева).
  - Митигация: инкрементальная синхронизация, rate limiting, кеширование; возможность ограничить синхронизацию заданными корнями (selected roots).
- i18n в будущем: риск “сломать” модель текста и идентификаторов при добавлении языков.
  - Митигация: хранить стабильный `doc_id` и список переводов (language codes) в метаданных; не завязывать идентификаторы команд на локализованные заголовки.

## Non-Goals
- Полноценный глобальный поиск по KB в UI (пока только дерево + выбор DocSet).
- Автоматическое сопоставление “какой раздел KB для какой схемы драйвера” (это ручная ответственность администратора).
- Попытка парсить “всю KB” — только выбранные разделы/DocSets (ориентир: CLI/Designer/ibcmd).
