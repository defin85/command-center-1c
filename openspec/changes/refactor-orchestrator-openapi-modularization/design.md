## Context
Текущий orchestrator-контракт хранится в одном файле `contracts/orchestrator/openapi.yaml` большой длины. Редактирование монолита повышает риск синтаксических ошибок и конфликтов, а также усложняет обзор изменений.

Одновременно текущий toolchain (валидация, генерация proxy routes и клиентов, breaking-changes check) жёстко ориентирован на единый файл `contracts/orchestrator/openapi.yaml`. Поэтому переход на модульные исходники должен сохранять единый bundle-артефакт как совместимый вход для существующих скриптов.

## Goals / Non-Goals
- Goals:
  - Перейти на модульные OpenAPI исходники для удобства сопровождения.
  - Сохранить единый bundle-файл `contracts/orchestrator/openapi.yaml` для текущих генераторов/валидаторов.
  - Сделать процесс детерминированным и проверяемым (bundle не должен устаревать незаметно).
- Non-Goals:
  - Менять API-поведение и схему данных из-за самой миграции структуры файлов.
  - Переписывать все contract-скрипты "с нуля" или менять стек генераторов без необходимости.

## Decisions
- Decision: Ввести двойной слой хранения контракта.
  - Source of edits: `contracts/orchestrator/src/**` (модульные файлы).
  - Source of tooling compatibility: `contracts/orchestrator/openapi.yaml` (bundle-артефакт).

- Decision: Bundle должен быть обязательным шагом перед validate/generate.
  - Скрипты должны либо собирать bundle автоматически, либо явно падать с понятной ошибкой "bundle устарел".

- Decision: Breaking-change check выполнять по собранным bundle-файлам.
  - Это снимает риск поломок на относительных `$ref` при сравнении через временные файлы.

- Decision: API parity при миграции контролировать автоматически.
  - Проверять, что ключевые структуры (`paths`, `operationId`, `components/schemas`) сохраняются без нецелевых изменений.

## Alternatives considered
- Оставить монолит `openapi.yaml` без изменений:
  - Отклонено: не решает проблему сопровождения и роста ошибок.
- Перевести tooling на чтение модульных файлов напрямую:
  - Отклонено на первом этапе: потребуется больше изменений в нескольких скриптах и повышается риск регрессий.
- Хранить только source-модули и не коммитить bundle:
  - Отклонено: ломает совместимость текущего pipeline и локального workflow.

## Risks / Trade-offs
- Риск: Появится дополнительный шаг сборки bundle.
  - Mitigation: Автоматизировать шаг в существующих скриптах и CI-check.
- Риск: Возможен дрейф между source и bundle.
  - Mitigation: Добавить обязательную проверку актуальности bundle с fail-fast ошибкой.
- Риск: Ошибки в разбиении могут случайно менять API.
  - Mitigation: Прогон validate + breaking-check + regression сравнение ключевых секций.

## Migration Plan
1. Подготовить модульный layout `contracts/orchestrator/src/**`.
2. Перенести текущий монолит в модульные файлы, сохранить все `operationId` и схемы.
3. Добавить/включить bundle script.
4. Обновить validate/generate/check-breaking скрипты на bundle-first режим.
5. Обновить документацию.
6. Прогнать проверки и зафиксировать новый workflow.

## Rollback Plan
- В случае проблем можно временно вернуться к редактированию единого `contracts/orchestrator/openapi.yaml`, отключив обязательность source-модулей в скриптах.
