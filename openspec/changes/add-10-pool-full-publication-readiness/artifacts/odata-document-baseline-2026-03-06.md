## Runtime Constraints Derived From Research
### Current policy slot is sufficient for first baseline
Текущий runtime уже поддерживает нужную точку подключения policy:
- primary: `edge.metadata.document_policy`
- fallback: `pool.metadata.document_policy`

Подтверждающие места:
- `orchestrator/apps/intercompany_pools/document_policy_contract.py`
- `orchestrator/apps/intercompany_pools/document_plan_artifact_contract.py`

Для первого full-run baseline менять storage model не требуется.

### Entity-only completeness profile is insufficient for production-like rollout
Текущий `completeness_profiles.entities.<entity_name>` недостаточен для полноценных prod-like BP 3.0 сценариев.

Причина:
- один и тот же `Document_ПоступлениеТоваровУслуг` имеет несколько валидных вариантов заполнения (`Товары` vs `Услуги`);
- один и тот же `Document_РеализацияТоваровУслуг` может отличаться между базами (`АгентскиеУслуги` vs `Услуги`).

Следствие:
- для production expansion потребуется variant-aware policy, минимум по признаку:
  - `entity_name + operation variant`
  - например через `ВидОперации` или явный policy profile selector на edge/run уровне.

## Verifier Note
Текущий verifier/complete-check трактует `0` и `false` как допустимо заполненные значения.

- для completeness нельзя считать числовой ноль автоматически "пустым".
