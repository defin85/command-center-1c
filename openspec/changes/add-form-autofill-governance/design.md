## Context
The browser warning class is not a React-only problem. It is produced from final DOM and can be introduced by:
- raw native form fields;
- AntD `Input`/`Input.Password`/`Input.TextArea`;
- AntD composite controls such as `Select`, `AutoComplete`, `DatePicker` and internal search inputs;
- platform form shells rendered through portals in modals/drawers;
- dynamic/schema-driven forms.

Existing repo surfaces already provide the right enforcement anchors:
- `frontend/eslint.config.js` contains the local `ui-platform-local` governance plugin.
- `frontend/src/uiGovernanceInventory.js` classifies route and shell surfaces.
- `frontend/tests/browser/form-field-ids.spec.ts` already checks duplicate ids and missing id/name for a small page set.
- `validate:ui-platform` already includes `npm run lint` and the broader UI platform browser gate.

External behavior to align with:
- HTML `autocomplete` is a browser hint available on `form`, `input`, `textarea` and `select`.
- Chrome DevTools exposes form/autofill issues in the Issues panel and Chrome DevTools Protocol has `Audits.checkFormsIssues`.
- AntD `Form.Item name` participates in generated field identity, but vendor composite controls may still render internal DOM fields that need runtime allowlisting rather than blind lint failures.

## Goals / Non-Goals
- Goals:
  - Make user-facing form field intent explicit and checked-in.
  - Catch new missing `id`/`name`/`autoComplete` regressions at lint time where static analysis is reliable.
  - Catch generated DOM regressions at browser-test time where static analysis is insufficient.
  - Keep false positives bounded for AntD composite/internal fields.
- Non-Goals:
  - Full historical remediation in the first task before enforcement exists.
  - Vendor DOM rewrites.
  - Password-manager behavior testing.

## Decisions

### 1. Field intent is the source of truth
Every governed user-facing form field must have an explicit intent that answers:
- whether the field is browser-autofillable;
- which autocomplete token is expected when it is autofillable;
- whether the field is a domain/operator control where autofill should not be offered;
- whether an exception is a vendor/internal composite input.

Candidate checked-in vocabulary:
- `username`
- `email`
- `current-password`
- `new-password`
- `one-time-code`
- `search`
- `domain-identifier`
- `domain-control`
- `not-autofillable`
- `vendor-internal`

Implementation may express this either through a platform wrapper prop such as `fieldPurpose`, or through inventory metadata for legacy/generated surfaces where wrapper migration is not yet feasible.

### 2. Static lint catches authored JSX contracts
The local `ui-platform-local` plugin should add form-field rules instead of introducing a parallel lint stack.

Static lint should fail for governed source modules when:
- native `input`, `textarea`, `select` or AntD `Input`/`Input.Password`/`Input.TextArea` appears without stable `id` or `name` unless it is wrapped by a known platform field primitive;
- a field whose `id` or `name` indicates username/email/password/OTP/search lacks an explicit `autoComplete`;
- credential fields use generic `autoComplete="off"` instead of `username`, `current-password`, `new-password` or `one-time-code`;
- a `Form.Item` with a user-facing field lacks `name` and lacks an explicit non-field/display exception.

Static lint should not attempt to fully reason about runtime DOM generated by AntD composite controls. It should instead require an explicit field intent at the authored component boundary.

### 3. Browser audit catches generated DOM and portals
`test:browser:forms` should become the runtime proof surface for the final DOM.

It should inspect:
- visible/enabled `input`, `select`, `textarea` fields;
- fields inside AntD portal containers for modal/drawer forms;
- duplicate ids within the page and within forms;
- missing `id`/`name`;
- fields with autofill-like `id`/`name` but no `autocomplete`;
- Chrome DevTools Protocol form issues via `Audits.checkFormsIssues`, if available in the Playwright browser runtime.

The browser audit must report actionable selectors and owning route/surface context, not only anonymous "violating node" counts.

### 4. Allowlist is checked-in and narrow
Some AntD internals are not meaningful user-facing fields. Those exceptions should be:
- centralized in a checked-in allowlist;
- scoped by component/surface/test selector;
- justified with a reason;
- blocked from broad wildcard matching.

An allowlist entry should expire or be tied to a migration task when it covers a real user-facing field rather than a vendor internal.

### 5. Adoption is staged
The first implementation should establish the policy, lint rule, browser audit and a pilot remediation surface. It should not require fixing all historical forms before any enforcement can merge.

Recommended rollout:
1. Enforce on platform primitives and newly changed platform-governed modules.
2. Add browser audit coverage for current high-traffic pages and modal/drawer open paths.
3. Remediate credential and login/admin connection fields first.
4. Expand enforcement to all platform-governed route and shell modules via inventory.

## Alternatives Considered

### Option A: Browser-only audit
Pros:
- sees final DOM, including AntD internals.

Cons:
- failures arrive late;
- hard to map to source component;
- easy to paper over with broad runtime allowlists.

Decision: insufficient alone.

### Option B: ESLint-only rule
Pros:
- fast and source-local.

Cons:
- cannot reliably see final DOM from AntD composite controls and portals;
- spread props/render props/schema-driven forms reduce precision.

Decision: useful but incomplete.

### Option C: Platform wrapper only, no lint
Pros:
- clean API for new work.

Cons:
- legacy and direct AntD usage can keep drifting silently;
- no blocking proof that the wrapper is used in governed surfaces.

Decision: wrapper should be the primary authoring path, backed by lint/browser gates.

## Risks / Mitigations
- Risk: false positives on AntD generated internal fields.
  - Mitigation: runtime allowlist scoped to vendor/internal controls and static intent at authored boundary.
- Risk: overuse of `autoComplete="off"` hides the problem but does not improve credential behavior.
  - Mitigation: lint forbids generic `off` for recognized credential fields.
- Risk: dynamic/schema-driven fields cannot be reasoned about by simple AST rules.
  - Mitigation: require schema-level field purpose metadata or platform wrapper generation.
- Risk: browser audit becomes slow if it opens every route and every modal.
  - Mitigation: start with `test:browser:forms` pilot paths and later shard through existing UI platform browser families.
