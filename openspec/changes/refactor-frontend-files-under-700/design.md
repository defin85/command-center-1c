# Дизайн: декомпозиция крупных frontend файлов

## Принцип разбиения
Разносить по ответственности:
- **страница** → секции/вкладки/панели (components)
- **логика данных** → хуки (hooks) и слой запросов (api/queries)
- **табличные конфиги/колонки** → отдельные модули (tables/columns)
- **валидации/форматирование** → utils

## Структура
Предпочтительно располагать подмодули рядом с владельцем:
- `frontend/src/pages/<Page>/components/*`
- `frontend/src/pages/<Page>/hooks/*`
- `frontend/src/pages/<Page>/utils/*`

Если компонент/хук переиспользуемый — перемещать в `frontend/src/components/*` или `frontend/src/hooks/*`.

## Совместимость
- Стараться не менять внешние импорты в большом объёме: где нужно — добавлять `index.ts` re-export.
- Не менять поведение: только перенос кода.

## Риски
- Циклические импорты при вынесении утилит и хуков.
- Случайные изменения порядка инициализации (особенно в больших страницах).
- Поломка тестов из‑за перемещения селекторов/`data-testid`.

## Контроль
После каждого “крупного” файла запускать `lint` + релевантный набор тестов, а не только в конце.
