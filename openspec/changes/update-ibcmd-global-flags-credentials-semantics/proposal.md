# Proposal: update-ibcmd-global-flags-credentials-semantics

## Почему (business value)
Сейчас `ibcmd` в UI/каталоге команд поддерживает только малую часть “общих параметров” (раздел 4.10.2 документации 1С), из-за чего:
- пользователи не видят и не могут управлять большинством driver-level флагов (`--temp`, `--log-data`, `--system`, `--db-path`, и т.д.);
- система не понимает семантику credential-флагов (что является DBMS creds, что является infobase creds), что усложняет безопасную инъекцию и маскирование;
- появляется риск конфликтов stdin (например `--request-db-pwd`) и неустойчивых интерактивных сценариев.

Цель: сделать `ibcmd` полностью schema-driven для driver-level опций, добавить типовую модель “credential bindings” и безопасную политику `service`‑аутентификации в ИБ для ограниченного набора safe‑команд (например extensions list/sync), без интерактива.

## Что меняется (scope)
- Расширяем `ibcmd.driver_schema` так, чтобы он отражал **все** флаги из общего раздела документации `ibcmd` (4.10.2 “Общие параметры”), включая алиасы.
- Для флагов, связанных с кредами, добавляем машиночитаемую семантику (какой флаг соответствует `db_user/db_password/ib_user/ib_password`), а также `sensitive`/маскирование.
- Стандартизируем: `db_pwd` хранится/передаётся как argv‑флаг (`--db-pwd` / `--database-password`), и **не** используем stdin‑флаг `--request-db-pwd (-W)` в guided‑режиме.
- Добавляем выбор стратегии `ib_auth` (`actor|service|none`) для `ibcmd_cli`:
  - `actor`: per-user mapping (как сейчас, по `created_by`);
  - `service`: сервисный пользователь ИБ (один на базу) только для allowlist команд и только для разрешённых ролей/permission;
  - `none`: только там, где ИБ‑аутентификация не нужна.
- Инжект кредов в worker производится через argv‑флаги `--user/--password` и `--db-user/--db-pwd` (без pty/expect).

## Невходит в scope (out of scope)
- Поддержка stdin‑режимов для DBMS пароля (`--request-db-pwd`) в guided‑режиме.
- Автоматическое создание сервисного пользователя в ИБ: пользователь должен существовать в ИБ и иметь минимально необходимые права.
- Расширение allowlist до dangerous команд.

## Impact
- UI: `DriverCommandBuilder` сможет рендерить полный набор driver options для `ibcmd` из schema.
- API: preview/plan/bindings смогут корректно маскировать креды и отражать provenance.
- Worker: добавляется стабильная инъекция creds для одновременной работы с СУБД + ИБ через argv.
- Безопасность: `service`‑стратегия доступна только для ограниченных команд/ролей и логируется как provenance без раскрытия секретов.
