# Change: Целевой лимит размера исходных файлов (700 строк) + отчёт

## Why
В проекте есть крупные исходные файлы (тысячи строк), из‑за чего:
- сложнее проводить ревью и локальные правки без побочных эффектов;
- хуже работает навигация по коду и повторное использование логики;
- LLM/агентам неудобно работать с контекстом (слишком большой файл → низкая точность изменений).

Цель — сделать кодовую базу более модульной и удобной для сопровождения, с ориентиром на верхнюю границу размера файла.

## What Changes
- Вводим проектное правило: **цель** — каждый исходный файл проекта держать в пределах **700 строк** (`wc -l`) за счёт декомпозиции по модулям.
- Формализуем, что входит в проверку и что исключается:
  - включаем: `*.ts`, `*.tsx`, `*.py`, `*.go` **включая тесты**;
  - исключаем: автогенерацию (например `frontend/src/api/generated/**`), архивы (`**/archive/**`), миграции Django (`**/migrations/**`) и прочие каталоги, где “разнос” нецелесообразен.
- Добавляем “мягкий” инструмент отчёта (скрипт/команда), который показывает список файлов >700 строк и динамику, но **не ломает CI** на первом этапе.
- Документируем правило и рабочий процесс (как читать отчёт, как планировать разнос по модулям).

## Non-Goals
- Не делаем жёсткий gate (CI fail) на превышение 700 строк (пока это “цель”, не строгий стандарт).
- Не выполняем сам рефакторинг кода в этом change (это делается отдельными change’ами по подсистемам).

## Impact
- Docs: добавится короткое описание правила и подхода.
- Tooling: появится команда для отчёта по крупным файлам.
- Последующие изменения: серия рефакторингов (frontend/orchestrator/go-services) будут ссылаться на это правило.
