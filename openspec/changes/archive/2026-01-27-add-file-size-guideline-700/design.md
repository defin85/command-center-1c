# Дизайн: лимит 700 строк как цель и отчёт

## Ключевые определения
- **Размер файла**: количество строк по `wc -l`.
- **Исходные файлы**: `*.ts`, `*.tsx`, `*.py`, `*.go`, включая тесты.
- **Исключения**:
  - генерируемые клиенты/типы (`frontend/src/api/generated/**`, `frontend/src/api/generated-gateway/**` и аналоги);
  - архивы (`**/archive/**`);
  - миграции Django (`**/migrations/**`);
  - прочие каталоги, где рефакторинг нецелесообразен (фиксируется явным списком паттернов).

## Почему “мягко” (без CI fail)
Порог 700 строк вводится как практический ориентир (качество и удобство работы), но на первом этапе:
- в репозитории уже много больших файлов;
- жёсткий gate будет шуметь и тормозить доставку функционала.

Поэтому сначала — отчёт + план рефакторинга, затем — поэтапное уменьшение крупных файлов отдельными change’ами.

## Принципы рефакторинга (для следующих change’ов)
- Только декомпозиция по ответственности (модули/файлы), без изменения поведения.
- Максимально сохранять существующие публичные API (экспорты, импорт-пути) либо менять их локально и последовательно.
- Разнос по модулям без “прагм” и искусственных исключений.
- Рефакторинг идёт по подсистемам (frontend / orchestrator / go-services), чтобы PRы были обозримыми.
