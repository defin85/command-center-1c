# Change: Обновить UI профиля IBCMD (выбор offline флагов из driver schema)

## Почему
Текущий UI редактирования `IBCMD connection profile` на странице `/databases` автодобавляет offline‑строки (`config`, `data`) даже при пустом профиле. Из-за обязательной валидации `value` это создаёт ощущение, что флаги обязательны, и блокирует сохранение, пока пользователь не заполнит/удалит строки.

Дополнительно, пользователь может вводить ключи в неверном формате (например `--db-name` вместо `db_name`), что приводит к “тихому” игнорированию флага на этапе выполнения.

## Что меняется
- UI модалки `IBCMD connection profile` больше НЕ добавляет никакие offline флаги по умолчанию.
- Пользователь сможет добавлять offline флаги “из схемы” (effective `driver_schema` для `driver=ibcmd`), выбирая их из списка.
- При этом профиль остаётся **raw flags**: система не интерпретирует режимы и не навязывает обязательные комбинации; ответственность за корректность значений остаётся на пользователе.

## Область/границы
- Меняем только UI/UX экрана `/databases` для редактирования профиля базы.
- Не меняем контракт хранения профиля (`Database.metadata.ibcmd_connection` v2 raw flags: `remote`, `pid`, `offline`).
- Не добавляем обратно DBMS metadata fallback для `ibcmd` (он удалён в v2).

## Ожидаемый результат (DoD)
- При открытии модалки для базы с пустым/отсутствующим профилем в секции `offline` нет строк и нет ошибок валидации.
- Пользователь может выбрать offline‑ключ из driver schema и добавить его в профиль, затем задать значение и сохранить.
- UI предотвращает/объясняет ввод ключей вида `--...` (ключи должны задаваться без префикса `--`).

## Влияние и риски
- Меняется UX формы: исчезают “подсказочные” строки `config/data`. Риск: пользователи будут меньше видеть “частые” флаги. Компенсация: список “из схемы” позволяет быстро найти эти ключи.
- Требуется доступ к effective `driver_schema` в UI `/databases` (используем существующий endpoint каталога команд).
