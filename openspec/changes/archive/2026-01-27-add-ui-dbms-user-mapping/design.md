# Дизайн: UI для DBMS user mapping

## Цели
- Дать администратору простой UI для настройки DBMS creds (per database) без необходимости ручного вызова API.
- Сохранить security-инварианты: пароль не выводится, DBMS creds не попадают в `DriverCommandBuilder`/shortcuts.

## Размещение в UI (предлагаемый вариант)
Добавить секцию/вкладку “DBMS users” на странице RBAC (рядом с существующим управлением Infobase users), так как:
- управление маппингами — административная функция;
- уже есть паттерны форм/таблиц/модалок, работающих с per-database mapping.

## Модель данных и режимы
Поддерживаем два типа записей:
- **actor mapping**: запись связана с конкретным CC пользователем (обязателен `user_id`), содержит `db_username` и “password configured”.
- **service mapping**: запись не связана с CC пользователем (`user_id` отсутствует) и помечена `is_service=true`.

UI должен явно разделять эти режимы (переключатель “Service account” / “Per-user mapping”) и валидировать ограничения:
- service mapping не может иметь `user_id`;
- actor mapping требует `user_id`;
- пароль вводится только через отдельную команду “Set password”.

## UX и безопасность паролей
- Поле пароля в списке отсутствует всегда.
- В таблице показываем только признак `db_password_configured` (например, badge “configured / not set”).
- “Set password” — модалка с вводом нового пароля (ввод скрыт), подтверждение действия.
- “Reset password” — отдельная кнопка, требующая подтверждения.

## Обработка доступа
- UI секция доступна только staff/admin пользователям.
- Если пользователь открыл страницу без прав (или права отозваны), UI должен корректно отработать 403 (показать сообщение/скрыть функциональность), без утечек данных.
