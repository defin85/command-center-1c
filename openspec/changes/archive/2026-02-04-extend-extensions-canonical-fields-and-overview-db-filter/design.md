## Контекст
В проекте используется двухслойная модель данных для расширений:
- `DatabaseExtensionsSnapshot.snapshot` хранит “нормализованный” список расширений (best-effort парсинг, может содержать больше полей).
- “Канонический” `extensions_inventory` используется как стабильная проекция для overview/drill-down и (в будущем) для tenant-scoped mapping.

Проблема: каноника слишком бедная, и `/extensions` не умеет ограничивать список расширений по конкретной базе без потери агрегатов.

## Цели
- Сделать канонический инвентарь достаточно выразительным для UI/диагностики (минимум безопасности и purpose).
- Добавить на `/extensions` фильтр “по базе” с семантикой:
  - фильтр ограничивает список имён расширений, но не изменяет логику агрегирования по всем доступным базам.

## Нецели
- Не менять контракт “сырых” результатов worker/driver.
- Не добавлять новые эндпоинты (если возможно) — расширяем существующий.

## Дизайн: канонический инвентарь
### Новые поля
`extensions_inventory.extensions[]` расширяется опциональными полями:
- `purpose: string | null`
- `safe_mode: boolean | null`
- `unsafe_action_protection: boolean | null`

Источник данных: нормализованный snapshot (best-effort). Если поле не извлечено/неприменимо — не включается или `null` (конкретика фиксируется в spec).

### Совместимость
Старые snapshots/клиенты остаются валидными:
- поля опциональны;
- агрегаты `/extensions` продолжают опираться на `name/is_active/version`, новые поля не влияют на вычисление статусов.

## Дизайн: фильтр по базе в /extensions
### Поведение
Добавляется `database_id` в `/api/v2/extensions/overview/`.

Алгоритм:
1) Проверить доступ пользователя к `database_id` (RBAC `view_database` + tenant context).
2) Загрузить snapshot выбранной базы:
   - если snapshot отсутствует/непарсится → вернуть пустой список (или `unknown`-поведение; фиксируется в spec).
3) Извлечь множество имён расширений из snapshot выбранной базы.
4) При формировании aggregated rows оставить только строки с именами из этого множества.
5) Все счётчики считаются по всем доступным базам (как сейчас).

### UI
На `/extensions` добавляется селектор базы (по доступным базам).
Фильтр:
- влияет только на верхнюю таблицу;
- в drawer по расширению фильтрация по базе — отдельная (не наследуется).

## Риски
- Производительность: на overview уже делается проход по доступным базам; добавление выборки snapshot одной базы — незначительно.
- Семантика “пустого” snapshot выбранной базы должна быть явной (пустая выдача vs unknown).
