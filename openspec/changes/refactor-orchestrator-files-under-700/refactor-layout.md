# Конвенция разбиения (orchestrator ≤ 700 строк)

Цель: реальная декомпозиция по модулям без изменения поведения и API.

## Views (API v2)

Уходим от монолитных `orchestrator/apps/api_v2/views/<domain>.py` к пакетам:

- `orchestrator/apps/api_v2/views/<domain>/__init__.py`:
  - публичные экспорты (совместимость для `urls.py` и тестов);
  - минимальный код: только импорты/реэкспорты и константы, если нужно.
- `orchestrator/apps/api_v2/views/<domain>/*.py`:
  - группы эндпоинтов и хелперов по ответственности (например, `read.py`, `write.py`, `audit.py`, `helpers.py`).

Принцип совместимости:
- `from apps.api_v2.views import <domain>` должен продолжать работать;
- атрибуты, используемые в `urls.py` и тестах, должны быть доступны как `views.<domain>.<name>` через `__init__.py`.

## Services

Для крупных модулей сервисов используем “тонкую” точку входа:

- оставить исходный модуль (например, `.../services.py` или `operations_service.py`) как совместимый фасад;
- вынести реализацию в соседние подмодули `*_impl.py`/`*_helpers.py`/подпакеты и реэкспортировать публичные функции/классы из фасада.

## Tests

Для тестов предпочтительно деление по сценариям:

- auth/permissions
- happy path
- validation/errors
- audit/rollback

Имена файлов: `test_<domain>_<scenario>.py` или `test_<feature>_<scenario>.py`.

## Риски и меры

- Порядок импортов и декораторы DRF/Django: в `__init__.py` держим только импорты, без побочных эффектов.
- Циклические импорты: выносить общие DTO/утилиты в отдельные `helpers.py`/`types.py` внутри домена.
