# Дизайн: декомпозиция крупных Python модулей в orchestrator

## Структура views
Цель — уйти от монолитных `views/*.py` к пакетам:
- `views/<domain>/__init__.py` — публичные экспорты, совместимость
- `views/<domain>/<area>.py` — группы эндпоинтов/хелперов по ответственности

Такой подход:
- уменьшает размер файлов;
- делает точки входа явными;
- снижает риск конфликтов при параллельной работе.

## Контроль совместимости
- Обновлять роутинг минимально: менять импорт‑пути на пакетные, сохраняя URL/имена view функций.
- Выносить только код; логика и ответы не меняются.

## Риски
- Скрытые побочные эффекты от порядка импортов (особенно у DRF/декораторов).
- Циклические импорты между подмодулями.

## Подход к тестам
Разделять “длинные” тесты по сценариям:
- auth/permissions
- happy path
- validation/errors
- audit/rollback
