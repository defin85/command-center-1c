# Gemini Context: CommandCenter1C

Этот документ содержит ключевую информацию о проекте CommandCenter1C, его архитектуре и способах работы с ним.

## 📋 Обзор проекта

**CommandCenter1C** — это централизованная микросервисная платформа для управления и выполнения массовых операций с более чем 700 базами данных 1С:Бухгалтерия 3.0.

- **Цель:** Обеспечить параллельную обработку данных, мониторинг в реальном времени и гибкую настройку операций через систему шаблонов.
- **Технологии:** Проект использует полиглотный стек, включая Go, Python (Django), и React.
- **Инфраструктура:** Оркестрация сервисов выполняется с помощью Docker. Для развертывания используется Kubernetes, а для мониторинга — Prometheus и Grafana.

### Архитектура

Проект построен на микросервисной архитектуре:

- **Frontend:** `React` + `TypeScript` (порт `5173`) — пользовательский интерфейс.
- **API Gateway:** `Go` + `Gin` (порт `8180`) — единая точка входа, аутентификация и маршрутизация.
- **Orchestrator:** `Python` + `Django` (порт `8200`) — основная бизнес-логика, управление задачами.
- **Task Queue:** `Redis Streams` — асинхронное выполнение задач.
- **Go Workers:** Масштабируемый пул воркеров на `Go` для параллельной обработки операций в 1С.
- **1C Integration:** прямой доступ из Worker к RAS/OData.
- **Базы данных:** `PostgreSQL` (основная), `Redis` (очереди и кэш), `ClickHouse` (аналитика).

## 🚀 Ключевые команды

Для разработки используются специализированные скрипты, которые управляют гибридной средой (инфраструктура в Docker, приложения на хосте).

**Важно:** Всегда используйте скрипты из каталога `./scripts/dev/`, а не `docker-compose` или `make` напрямую.

### Основной цикл разработки

1.  **Запуск всех сервисов:**
    *   Эта команда выполняет "умный" запуск с автоматической пересборкой только измененных Go-сервисов.
    ```bash
    ./scripts/dev/start-all.sh
    ```

2.  **Проверка состояния сервисов:**
    ```bash
    ./scripts/dev/health-check.sh
    ```

3.  **Перезапуск сервисов (после внесения изменений):**
    *   Перезапускает все сервисы с "умной" пересборкой.
    ```bash
    ./scripts/dev/restart-all.sh
    ```
    *   Перезапуск одного конкретного сервиса:
    ```bash
    ./scripts/dev/restart.sh <имя_сервиса>
    # Пример: ./scripts/dev/restart.sh orchestrator
    ```

4.  **Просмотр логов:**
    ```bash
    ./scripts/dev/logs.sh <имя_сервиса>
    # Пример: ./scripts/dev/logs.sh api-gateway
    ```

5.  **Остановка всех сервисов:**
    ```bash
    ./scripts/dev/stop-all.sh
    ```

### Тестирование

Для запуска тестов используются цели из `Makefile`.

- **Запустить все тесты (Go, Python, Frontend):**
  ```bash
  make test
  ```
- **Запустить тесты для конкретной платформы:**
  ```bash
  make test-go
  make test-python
  make test-frontend
  ```

### Сборка

- **Собрать все бинарники Go:**
  ```bash
  make build-go-all
  ```
- **Собрать все Docker-образы:**
  ```bash
  make build
  ```

### Миграции базы данных

Миграции Django применяются через `Makefile`:
```bash
make migrate
```

## 📐 Соглашения по разработке

1.  **Гибридный режим:** Инфраструктура (`postgres`, `redis`) работает в Docker, а приложения (`orchestrator`, `go-services`) запускаются на хост-машине.
2.  **Использование скриптов:** Для управления средой разработки предпочтительно использовать скрипты из `./scripts/dev/`.
3.  **Порядок запуска:** Критически важен порядок запуска сервисов. Скрипты `start-all.sh` учитывают это.
4.  **Общий код:** Общий код для Go-сервисов находится в `go-services/shared/`.
5.  **Тестирование:** Все новые функции и исправления должны сопровождаться тестами. Целевой уровень покрытия — более 70%.
6.  **Коммуникация сервисов:** Фронтенд взаимодействует **только** с `API Gateway`. Прямые вызовы `Orchestrator` запрещены.
